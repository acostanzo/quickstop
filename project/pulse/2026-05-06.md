- Reviews used to glom every finding into a single `gh pr comment` wall-of-text. Real reviewers leave inline comments on specific lines under one review submission with a short overall verdict; v2.1 makes that the path of least resistance by locking a JSON contract between the formatter (LLM) and the poster (deterministic bash).

One review, N inline comments grouped under it.

New top-level runner at `tests/run-all.sh` walks the existing trailer-stripping suites plus the new post-review test.

## 21:30

T3 shipped: `/lintguini:lint` and `/lintguini:format` wired across all six languages, finding-shape contract locked at `path:line:rule:message`.

- `bin/lintguini-lint.sh` is the deterministic dispatcher. Per-language tool selection follows the rubric: `ruff check --output-format=json` for python, `biome check --reporter=github` for js/ts, `cargo clippy --message-format=json` for rust, `rubocop|standardrb --format json` for ruby (auto-picked from the provenance-marked config — `standard.yml` wins when both exist because configure leaves stale `.rubocop.yml` in place per its docs), `golangci-lint run --out-format=json` for go. No fallback to a "second-best" linter — ADR-008's authority pin would otherwise make `/lintguini:configure --strict` followed by `/lintguini:lint` grade against a different rubric than the one configure wrote.
- Each tool's native output is projected to the locked shape via per-language jq filters (or sed for biome's GitHub annotations). Rule names are preserved verbatim — `F401`, `lint/correctness/noUnusedVariables`, `clippy::needless_return`, `Style/FrozenStringLiteralComment`, `errcheck` — because the rule code is what consumers grep for in suppression comments. Embedded newlines in tool messages are collapsed to single spaces so each finding stays one line. Findings sort by `(path, line, rule, message)` for triple-run determinism.
- `bin/lintguini-format.sh` runs both modes from the same dispatch table. Apply mode runs the per-tool check pass first to enumerate which files WILL change, then runs the apply pass; the pre-apply file list becomes the `formatted <path>` lines on stdout. Two tool invocations per language is the cost of a tool-agnostic per-file output contract; formatters are idempotent so the second pass is bounded. `--check` mode emits `would format <path>` lines and exits 1 when any file would change (CI-friendly).
- Polyglot output: when more than one configured language is in scope, stdout is sectioned by language with a `# <language> — N findings` (lint) or `# <language> — N files [would be] formatted` (format) header. Single-language runs (`--language <lang>` or only one configured language) emit no header. Header lines start with `^# ` so M5's `score-lint-pass-rate.sh` can grep them off when counting findings — documented inline in both SKILL.md files.
- Empty-scope handling: a detected-but-not-lintguini-configured language emits `<lang>: not configured (run /lintguini:configure --language <lang>)` to stderr and exits 0. Empty-scope is not a failure — it's the "nothing to lint here" signal. Tool-missing emits `<lang>: <tool> not on PATH (install <tool>)` to stderr and exits 3 (tooling error). Both messages are greppable for downstream tooling.
- Hit a real bug in format: `format_lang` calls `files_<tool>` via `$()` inside another `$()` (main's `out=$(format_lang)`). When `tool_missing`'s `exit 3` ran from inside `files_<tool>`, the rc was swallowed by the inner-most subshell before reaching main — script continued and exited 0 with no findings. Fixed by pre-flighting tool availability in main's loop, one level above `format_lang`. Documented the contrast against lint (where `lint_<lang>` IS the outermost subshell so `tool_missing` propagates one level up to main correctly) so a future author doesn't re-introduce the layering bug.
- Tests are 33 checks for lint and 36 for format, all driven by mock binaries staged on PATH per case. Mocks emit each tool's known native output (ruff JSON array, biome `::error` GitHub annotations, cargo NDJSON `compiler-message` lines, rubocop/standardrb `.files[].offenses[]` JSON, golangci-lint `.Issues[]` JSON, gofmt `-l` file list); the script's per-language projection is the contract under test. Coverage hits configured+violations per language, configured+clean, not-configured (greppable empty-scope), tool-missing (greppable hint + rc=3), polyglot section headers, `--language` scoping (no header), triple-run determinism, argument errors (rc=2), apply failure (rc=4), section-header singular/plural, and check vs. apply verb shape.

## 21:36

T4 shipped: `/lintguini:fix` wired across six languages with the mechanical-vs-semantic split. M4 done; M5 (conditional audit scorers + plan close + version bump) is the only remaining ticket.

- `bin/lintguini-fix.sh` is the deterministic dispatcher. Three modes from one entry point: default (read-only diff preview, exit 1 if anything to fix), `--apply` (mutates the working tree, emits `fixed <path>` per touched file, exit 0), `--apply-semantic` (unsafe-fix diff to stdout, never writes — the user pipes into `git apply` themselves). `--apply` and `--apply-semantic` are mutually exclusive (exit 2). Mirrors `/inkwell:tidy`'s split — semantic changes never land without human review.
- Per-language dispatch follows the rubric strictly. Python uses ruff's native flags (`--fix --diff` / `--fix` / `--fix --unsafe-fixes --diff`) — clean three-mode coverage. JS/TS go through copy-and-diff for both preview shapes (biome lacks a native diff mode); apply runs `biome check --write` directly. Rust uses copy-and-diff for the safe-preview path (cargo clippy --fix --dry-run reliability is patchy across versions) and empty-scopes `--apply-semantic` (clippy has no safe/unsafe split). Ruby splits between rubocop (copy-and-diff for both preview shapes; `-a` safe + `-A` autocorrect-all for semantic) and standardrb (copy-and-diff safe; semantic empty-scoped because standardrb's whole point is no rule-by-rule debates). Go uses native `gofmt -d` / `gofmt -w` for safe; semantic empty-scoped (gofmt is formatter-only and golangci-lint's --fix --dry-run support is too version-fragile to depend on in v1).
- Copy-and-diff is the universal fallback for tools without a native preview mode: snapshot REPO_ROOT to a tempdir (skipping `.git`, `node_modules`, `vendor`, `target`, `dist`, `build`), run the fix tool against the copy, then `diff -u --label "a/<path>" --label "b/<path>"` each file in the original against the copy. Working tree untouched. Documented inline in the bin so a future contributor knows when to reach for it (and importantly, when not to — copy_and_diff is the right call only when (a) the tool can't preview without writing, or (b) the tool's preview mode is too version-fragile).
- Output is normalised to standard `--- a/<path>` / `+++ b/<path>` unified-diff headers regardless of which underlying tool emitted the diff (ruff's bare-path output is rewritten via an awk filter; copy_and_diff already emits the prefix natively). The user can pipe lintguini-fix's stdout straight into `git apply` for any of the six languages without per-language post-processing.
- Polyglot output is sectioned with `# <lang> — N file[s] with [semantic ]fixes available` (preview) or `# <lang> — N file[s] fixed` (apply) headers. The `# ` prefix doubles as a unified-diff comment line — git apply ignores it — so the polyglot output remains pipe-clean. Single-language runs emit no header.
- Empty-scope handling. Detected-but-not-configured language → `<lang>: not configured (run /lintguini:configure --language <lang>)` to stderr, exit 0. Tool not on PATH → `<lang>: <tool> not on PATH (install <tool>)` to stderr, exit 3. No fixable issues found → `<lang>: no fixes available` to stderr, exit 0. The empty-scope-by-design semantic combos (rust, go, standardrb) emit their own explanatory stderr (`<lang>: --apply-semantic empty-scope (<tool> has no safe/unsafe split)`) and the driver suppresses the generic `no fixes available` to avoid double-messaging — `is_empty_scope_by_design` predicate guards this.
- Same subshell-layering trap as T3's format script: `tool_missing`'s `exit 3` from inside `fix_<lang>` would be swallowed by the inner-most subshell (out=$(fix_lang)) before reaching main. Pre-flighting tool availability in main's loop, one level above `fix_<lang>`, mirrors the format.sh fix. Documented inline. Notably, semantic-mode empty-scope languages (rust/go and standardrb) skip the tool check since they never invoke the tool — keeps the pre-flight from rejecting a perfectly-valid `/lintguini:fix --apply-semantic` on a rust repo where cargo isn't installed.
- `skills/fix/SKILL.md` is the LLM-shaped half. Frontmatter (`allowed-tools: Read, Bash, Glob`, `argument-hint: [--language <lang>] [--apply | --apply-semantic]`); body documents the three-mode contract, the per-language dispatch table, the safe/semantic boundary (deferring to each linter's own definition rather than a parallel notion), and the copy-and-diff fallback inline so the contract is greppable from the surface a user reads. Mirrors the lint/format SKILL conventions: stdout is the contract, stderr carries empty-scope/tool-missing diagnostics, exit codes match the bin's.
- Tests are 36 checks across 15 functions, hitting all 10 cases from the brief: read-only diff (with bare-path normalisation), apply emitting `fixed <path>`, configured+clean emitting `no fixes available`, semantic mode (unsafe-mode diff with sha256 working-tree-untouched check), standardrb semantic empty-scope, mutex flags (rc=2), apply idempotency (marker-file-driven mock returns nothing-to-fix on second call), triple-run determinism on read-only AND semantic (apply mutates state by design so it's excluded), polyglot (python native + javascript copy-and-diff in one run), and the load-bearing copy-and-diff fallback test (biome mock writes to cwd; script snapshots, mock writes to copy, original untouched, emitted diff carries `a/`/`b/` headers and reflects the mock's edit).
