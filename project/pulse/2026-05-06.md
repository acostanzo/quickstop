# Pulse — 2026-05-06

## 13:30

Released commventional 2.1.0 — inline review comments via grouped review submission.

- Reviews used to glom every finding into a single `gh pr comment` wall-of-text. Real reviewers leave inline comments on specific lines under one review submission with a short overall verdict; v2.1 makes that the path of least resistance by locking a JSON contract between the formatter (LLM) and the poster (deterministic bash).
- `agents/review-formatter.md` now emits a single JSON document — `{ verdict: { event, body }, comments: [{ path, line, side, label, subject, discussion? }, ...] }` — instead of human-readable text. Output spec is JSON-only: no prose, no fences, no commentary. The poster pipes the agent's stdout straight into `jq`.
- New `bin/commventional-post-review.sh` consumes the JSON on stdin, validates shape (path/line/label/subject required per comment; line must be numeric; event ∈ {COMMENT, APPROVE, REQUEST_CHANGES}; side ∈ {LEFT, RIGHT}), resolves head SHA via `gh pr view`, builds the GitHub review-API payload, and submits via a single `gh api POST repos/{owner}/{repo}/pulls/{n}/reviews --input -`. One review, N inline comments grouped under it.
- `skills/commventional/SKILL.md` "Code Reviews" step 3 rewritten and a "Response contract — locked" subsection added inline so a future contributor can wire against the field shape without re-deciding.
- Test surface: `test-fixtures/post-review/` ships `sample-review.json` (full review, 3 comments), `sample-review-missing-path.json` (validation rejection), `sample-review-empty.json` (verdict-only). `post-review-dryrun.test.sh` covers all four cases (rendering, rejection, empty, triple-run determinism). `--head-sha` flag bypasses live `gh pr view` so the test surface runs without network or gh auth. New top-level runner at `tests/run-all.sh` walks the existing trailer-stripping suites plus the new post-review test.
- Smoke pass against PR #102 produced the expected `gh api` invocation referencing `repos/acostanzo/quickstop/pulls/102/reviews` with a head SHA matching `gh pr view 102 --json headRefOid`. Comment body preserves the conventional-comments shape on the wire (`label: subject\n\ndiscussion`).
- Out of scope: APPROVE/REQUEST_CHANGES heuristics (the agent decides; the poster passes through), `commit-crafter` changes, the trailer-stripping skills, `user-invocable: false` on the auto-skill.

The `mid` snapshot fixture in `test-fixtures/snapshots/` is currently broken on main — the pinned SHA's worktree carries 10 trailer-bearing commits while the snapshot pins 17. Pre-existing, unrelated to this PR; flagged separately.

## 20:46

T2 shipped: `/lintguini:configure` wired, standardrb detection fixed, A1 round-trip ready for M5 to wire.

- `bin/lintguini-configure.sh` is the deterministic half — flag parsing, language scoping (`--language`), band selection (`--strict | --lenient | --minimal`, mutually exclusive), Ruby tool resolution (`--ruby-tool` flag or Gemfile auto-detect — `gem "standard"` → standardrb, otherwise rubocop), and per-file write/merge/copy with byte-equal idempotency. Pure bash + jq + python3 (tomllib for parsing, regex-based preamble + section stripper for TOML merging). Exit codes 0/2/3/4 mirror the brief.
- `skills/configure/SKILL.md` is the LLM half — frontmatter (`allowed-tools: Read, Write, Edit, Bash, Glob`), argument-hint matching the bin's flag surface, body documenting the dispatch shape, output verbs (`wrote` / `updated` / `skipped (already configured)` / `would-*` / `warning`), and the per-language file map. Mirrors the inkwell-doc + inkwell-doc-resolve.sh skill+bin split.
- TOML merge approach: provenance-aware section stripper in Python (`strip_toml_managed`). Drops the lintguini preamble (provenance line + every comment-or-blank line until the next non-comment line — robust to wrapped band comments) AND any TOML sections matching the supplied prefix list (e.g., `tool.ruff`, `lints`). Runs through `python3` since we already require 3.11+ for tomllib in the audit path; awk wasn't expressive enough for the multi-line preamble case. Cycling strict → lenient → minimal → strict is byte-equal-idempotent on the second of any pair.
- JSON merging via `jq`. Always pipes through jq even on fresh writes so the formatter output matches between fresh-cp and merge-into-empty paths — a `cp` of the template followed by a re-merge would otherwise byte-diverge and break idempotency on the second run. Caught this on the first smoke pass; fixed before tests landed.
- CI wiring (`--ci`) detects in priority order: `.github/workflows/` → writes a fresh `lintguini-lint.yml` (most-conventional surface, follows the brief's "GitHub Actions for most repos" rule); `.gitlab-ci.yml` / `Makefile` / `lefthook.yml` / `.pre-commit-config.yaml` → appends a guarded block (idempotency via grep for the provenance line). No CI surface present → warns to stderr ("no CI surface detected; --ci skipped") and proceeds with local config writes. Never invents a CI surface from scratch.
- Standardrb-detection fix landed as its own atomic commit. `scorers/_common.sh::detect_primary_language` now treats `standard.yml` and `.standard.yml` as ruby markers alongside Gemfile / `*.gemspec` / `.rubocop.yml`. Without the fix the plural `detect_languages` (which already accepted `standard.yml` from T1) and the singular path would disagree on a standardrb-only repo: configure would scope to ruby, audit would empty-scope. Added a `ruby-standardrb` fixture to `scorers/tests/fixtures/formatter-presence/` to cover the case. Locked end-to-end snapshots are byte-equivalent (no existing fixture was standardrb-only) so the fix didn't trigger any re-locking.
