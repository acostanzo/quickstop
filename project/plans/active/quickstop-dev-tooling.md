# Quickstop Dev Tooling — Smith and Hone Optimization

## Why this plan exists

Smith and hone are quickstop's internal dev tools — used when authoring
or maintaining plugins in this repo, not shipped as marketplace plugins.
They encode the design principles every plugin in the marketplace
should follow.

Both predate ADR-004 (sibling composition contract), ADR-005 (sibling
skill conventions), and ADR-006 (plugin responsibility boundary). The
result: smith scaffolds plugins that don't know about pronto siblings,
don't carry an ADR-006 §1 README surface section, and offer no guard
against ADR-006 §2/§3 silent-mutation patterns; hone audits plugins
without checking sibling-shape compliance or hook-behavior boundaries.
Smith also silently defaults to MIT against
`.claude/rules/license-selection.md`'s "never default-pick" rule.

Before continuing Phase 2 sibling rollouts (PR 2b lintguini, PR 2c
towncrier audit extension) and the M-series legacy migrations (claudit,
commventional → observations[]), smith and hone need to be current with
our own conventions. This plan optimizes them.

## Scope

Two tickets, sequenced.

| Ticket | Scope | Depends on |
|---|---|---|
| **Q1 — Smith enhancements** | License question; sibling-shape branch in questionnaire; `pronto` block + `:audit` skill auto-creation; transitional parser agent scaffolding; ADR-006 §1 README "Plugin surface" section scaffolded for every plugin; no hook scaffolding without an explicit user-facing migration prompt; ADR-aware research targets | — |
| **Q2 — Hone enhancements** | New Pronto Compliance audit category; new `audit-pronto` subagent; new audit-boundary subagent that detects ADR-006 §1 surface omissions, §2 silent-mutation patterns (Scope A transitively along the call graph from hooks; Scope B without consumer opt-in), and §3 hook-invariant violations (Tier 1 enumerated paths only; Tier 2 variable targets emit human-review notes without auto-deduction); rubric rebalance; ADR-aware research targets | Q1 (shares the research-agent change) |

Both tickets are quickstop-internal — they touch `.claude/skills/smith/`,
`.claude/skills/hone/`, and `.claude/agents/research-*`. No marketplace
plugin changes.

## Out of scope

- **Shared research infrastructure between smith and hone.** The audit
  recommended deferring extraction until ≥3 callers exist. Today there
  are two, and the duplication is small. Refactor when a third caller
  appears (e.g. a future `forge` or a pronto-side preflight skill).
- **`:doctor` and `:fix` skill scaffolding.** ADR-005 §2 / §4
  reservations; no contract yet. Smith doesn't scaffold either.
- **Migrating existing marketplace plugins** to be sibling-shaped.
  Per-plugin work — the M-series in `phase-2-pronto.md` and the
  inkwell implementation tickets (2a1/2a2/2a3) cover those.

## Gating

Phase 2 PRs 2b (lintguini) and 2c (towncrier audit extension) **wait
for Q1+Q2 to land.** Inkwell implementation (the 2a1/2a2/2a3 tickets)
also waits — those tickets stay open in `tickets/open/`, but no
implementation begins until smith is current.

**Inkwell vs. dogfood.** The 2a1 ticket fully specifies its
deliverables (plugin.json, LICENSE, README, SKILL.md body). When
implementation begins, **2a1 is hand-implemented per its existing spec** —
not regenerated by smith. Re-running `/smith inkwell` against its spec
would either duplicate work or contradict the ticket's intentional
shape. The optimized smith's **first true dogfood is lintguini (PR
2b1)**, which doesn't carry a pre-written scaffold spec — the 2b1
ticket should be revised post-Q1 to say "scaffold via `/smith
lintguini` with role=sibling, dimension=lint-posture" rather than
hand-spec'ing the file tree. That revision is in 2b1's scope, not
this plan's.

The M-series migrations (M1 claudit, M3 commventional) can begin as
soon as Q2 lands — they're driven by hone's Pronto Compliance findings
against the shipped siblings (which Q2 detects via the
`recommendations.json` registry path even when the plugin doesn't yet
declare a `pronto` block in `plugin.json`).

## Acceptance for plan completion

- Q1 and Q2 merged.
- `/smith inkwell` (or another fresh sibling name) produces a
  sibling-shaped scaffold matching ADR-005 §1: `skills/audit/SKILL.md`
  present, wire-contract emission shape valid, transitional parser
  agent scaffolded.
- `/hone skillet` produces a Pronto Compliance score; skillet — the
  most ADR-005-conformant sibling — scores ≥85/100 in that category.
- `/hone claudit` and `/hone commventional` are detected as siblings
  via the `recommendations.json` registry path and produce Pronto
  Compliance findings; the findings exactly match the M-series work
  outlined in `phase-2-pronto.md` (missing `pronto` block in
  `plugin.json`, missing `skills/audit/SKILL.md` at the canonical
  path, no `observations[]` emission). Hone is the diagnostic that
  drives M1/M3.
